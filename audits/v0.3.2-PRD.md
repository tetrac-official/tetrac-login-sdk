# PRD — `@tetrac/login-sdk` v0.3.2 (audit reconciliation + remaining work)

**Target release:** `v0.3.2` (current branch `v0.3.2`; `package.json` is `0.3.1`)
**Basis:** [`audits/agent-ai.md`](./agent-ai.md) — two external AI security audits ("Audit A", "Audit B") —
reconciled against the **actual HEAD code** and the work already closed in
[`audits/v0.3.0-PRD.md`](./v0.3.0-PRD.md) / [`audits/opus-4.8-comprehensive-audit.md`](./opus-4.8-comprehensive-audit.md).
**Status at authoring:** code uses `@noble/hashes` + Web Crypto (no `crypto-es`); AES-256-GCM secret
encryption; 600k-default PBKDF2; memory-only vault; ed25519 signature auth (no `passkeyHash`).

> **Read this first.** `agent-ai.md` was written against a *generic* "SDK of this class," **not** this
> codebase — both audits say so explicitly ("implementation logic *typically* found in…"; "Exact line
> numbers are mapped to *standard SDK layout*"). As a result **the large majority of their findings are
> already fixed** in shipped code. §1 documents that reconciliation with file:line proof; §2 carries only
> the genuinely-open items the audits correctly (or accidentally) point at.

---

## 0. How to read this PRD

- §1 is the **audit reconciliation**: every finding from Audit A and Audit B with a verdict
  (`ALREADY SHIPPED` / `FALSE PREMISE` / `OPEN` / `OPTIONAL` / `N/A`) and the proof.
- §2 is the **actionable residue** — the small set of items the audits get right that are *not* yet done.
  Work items continue the existing numbering (WI-15/16 carry over from the v0.3.0 PRD; new items start WI-22).
- §3 covers **docs/release-eng**; §4 the **already-documented developer-owned residuals**.
- Each open WI gives a **proven problem** (file:line), a **design**, **files touched**, and an
  **acceptance test**.

### WI-15 — Real server-side WebAuthn assertion (Audit A-D, **High**, carried from v0.3.0)

**Problem (proven).** Biometric accounts never prove a hardware WebAuthn assertion to the server. The client
derives an ed25519 keypair from the PRF/gate secret (`src/client/authKey.ts:19-28`) and the server verifies
that *derived-key* signature (`src/server/signature.ts:45-58`). The server stores no credential COSE public
key and checks no `origin`, `rpIdHash`, `signCount`, or `userVerification`. Audit A finding D points here.

**Goal / design / files / acceptance:** unchanged from [`v0.3.0-PRD.md` WI-15](./v0.3.0-PRD.md) — register
verifies attestation and stores `{credentialID, publicKey, signCount, transports}`; login verifies the
assertion against the stored key incl. `origin`/`rpIdHash`/single-use challenge/UV/strictly-increasing
`signCount`. Add `@simplewebauthn/server` (dep of `/server`) + `@simplewebauthn/browser` (peer of `/client`).
Acceptance: derived-key-only login rejected; replayed/tampered/clone (`signCount`) assertions rejected.

### WI-16 — HKDF enc/auth key separation (Audit B Crypto-3, Med, carried from v0.3.0)

**Problem (proven).** The auth seed is `SHA-256("ttc-auth-v1:" + appKey)` (`src/client/authKey.ts:13-16`) —
the same high-entropy secret seeds the auth keypair *and* (as the app key) encrypts wallets. Audit B's
"credential-swap" rationale is wrong, but the underlying recommendation (derive purpose-specific subkeys via
HKDF) is sound and already partly realized elsewhere (`biometricUnlock.ts` HKDF wrap key).

**Design.** Replace the bare-SHA-256 seed with `HKDF-SHA-256(appKey, info)` using distinct labels:
`info="ttc-enc-v1"` for encryption, `info="ttc-auth-v1"` for the email-account auth seed. `@noble/hashes`
already ships `hkdf` (`@noble/hashes/hkdf.js`) — **no new dependency**.

**Files.** `src/core/crypto.ts`, `src/client/authKey.ts`. **Migration:** changes the derived ed25519 key for
email accounts → clean break, re-register (consistent with the maintainer's no-migration policy). Bundle with
WI-15. **Acceptance:** `encKey ≠ authValue` for the same secret; both deterministic; matches a known HKDF vector.

### WI-23 — Optional session→User-Agent binding (Audit B Auth-3, Low, optional) — ✅ DONE

**Problem (proven).** Sessions already expire (4h TTL) and revoke the prior token on each login
(`src/server/session.ts`), but there was no device/fingerprint binding. Audit B over-claimed
("persist indefinitely / non-CSPRNG") — both false.

**Decision — no separate `/revoke` endpoint.** The model is **single-active-session** (`issueSession`
revokes the prior token), and `POST /logout` already revokes the current/only token
(`routes.ts:logout`). A `/revoke` of the presented token would therefore duplicate `/logout`, and there
are no "other" sessions to revoke. The genuinely-additive part — UA binding — is what shipped.

**Shipped design.** `config.bindSessionToUserAgent` (default **false**). When on, `issueSession` stores
`publicKey|SHA-256(User-Agent)` and `verifySession` rejects (constant-time) a request whose UA hash
differs or is missing. Enforcement is **per-session** (keyed on whether a fingerprint is stored), so
disabling the flag never un-binds live sessions. UA encoded into the session value (no `UserData` change),
so the bare-`publicKey` format is preserved when the flag is off.

**Files (done).** `src/core/config.ts` (`bindSessionToUserAgent`), `src/core/crypto.ts` (`hashUserAgent`),
`src/server/session.ts` (encode/decode + threaded fingerprint), `src/server/routes.ts` (issue/verify
fingerprint helpers, 4 issue + 3 verify sites), `tests/session-fingerprint.test.ts`. Docs: README Security
model, `docs/CRYPTO_SPEC.md` §5.2. **Migration:** additive (default off). **Acceptance (met):** binding-off
keeps the bare-`publicKey` value + ignores UA; binding-on accepts matching UA, rejects different/missing UA;
a bound session stays enforced after the flag is disabled. Full suite green (179).

### WI-24 — `SECURITY.md` + threat model + crypto spec (Audit A & B docs, Low)

**Problem (proven).** No `SECURITY.md`, threat model, or crypto specification exists in the repo (verified).
**This is the root cause of the audits' inaccuracy** — with no published primitive/version spec, external
reviewers defaulted to assuming a generic CBC/`crypto-es` SDK and audited that instead of the real code.

**Design.** Add `SECURITY.md` (disclosure policy + supported-browser matrix — Web Crypto required, no IE11)
and `docs/CRYPTO_SPEC.md` documenting: KDF (`PBKDF2-HMAC-SHA256`, `securityLevel`→iterations, domain-separated
salt `SHA-256(appId:email)`), secret encryption (`AES-256-GCM`, `iv:ct+tag` b64url), auth model (ed25519
signature, no `passkeyHash`), vault model (memory-only, lifecycle locks), and the versioned `info` strings.
**Files.** `SECURITY.md`, `docs/CRYPTO_SPEC.md`, README link. **Migration:** docs-only. **Acceptance:** docs
present and linked; spec matches `crypto.ts`/`config.ts` constants (CI doc-lint optional).

### WI-25 — Optional Argon2id (memory-hard) KDF path (Audit B Crypto-2, Low, optional/future)

**Problem.** PBKDF2 (even at 600k) is GPU/ASIC-amenable; Argon2id is memory-hard. Not a fix (600k is
NIST-compliant), but a defense-in-depth option for low-entropy passkeys.

**Design.** Add `securityLevel`-adjacent opt-in `kdf: "pbkdf2" | "argon2id"` to `AuthConfig`; when `argon2id`,
derive via a WASM Argon2 (e.g. `hash-wasm`) as a **peer/optional** dep, pinning params per-user like
iterations are today. Keep PBKDF2 default. **Files.** `src/core/crypto.ts`, `src/core/config.ts`,
`src/core/types.ts` (`UserData.kdf`), `package.json`. **Migration:** opt-in per deployment; changing an
existing account's KDF is a re-register. **Acceptance:** an `argon2id` account derives deterministically and
round-trips encrypt/decrypt; PBKDF2 accounts unaffected. **Risk:** WASM bundle weight + SSR/edge support —
confirm before adopting.

### WI-26 — Wire `revealRequiresReauth` to a per-op re-auth ceremony (Audit B Vault-2, Low, optional)

**Problem (proven).** `revealRequiresReauth` is defined and defaults `true` but is **reserved/not enforced**
("v1 always re-auths", `src/core/config.ts:109-113,144`). Today the only re-auth gate is the 15s idle
auto-lock. Audit B asks for an explicit passkey re-prompt before `signTransaction`/`exportPrivateKey`.

**Design.** On reveal/export (and optionally sign), when `revealRequiresReauth` is true, require a fresh
`navigator.credentials.get({ userVerification: "required" })` (or app-key re-derivation) within a short cache
window (≤5 min) before releasing plaintext. Reuse `withDecryptedKey` (`wallet.ts:94-105`) for lifetime
minimization. **Files.** `src/client/authClient.ts`, `src/client/wallet.ts`, `src/react/useExportKey.ts`.
**Migration:** behavioral; gated by the existing (default-on) flag. **Acceptance:** export without a fresh
ceremony is rejected when the flag is on; succeeds with one; cached for ≤5 min.

## 5. Acceptance & sequence for v0.3.2

1. **WI-24 (docs) ships first and independently** — cheap, non-breaking, and removes the root cause of audit
   inaccuracy by publishing a spec to audit against.
2. **WI-15 + WI-16 land together** as the one breaking release (biometric assertion + HKDF separation), per
   the v0.3.0 plan; existing biometric/email accounts re-register.
3. **WI-23 / WI-25 / WI-26** are optional, additive (or flag-gated) and may land in any order or slip to a
   later minor.
4. Full suite stays green; `tsc --noEmit` clean; `npm audit` shows no new (non-Solana-transitive) advisories.

**Suggested order:** `WI-24 (docs) → WI-15 (+WI-16) [one breaking release] → WI-23/26 → WI-25 (if WASM fit
confirmed) → CHANGELOG → publish (--provenance) → demo bump`.
