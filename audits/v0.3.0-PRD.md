**Target release:** `v0.3.0` (`package.json` is already `0.3.0`)

**Status at authoring:** full suite green (**170/170**), `tsc --noEmit` clean, build clean.
**Supersedes:** the v0.2.2 PRD's *"backward-compat / versioned-migration"* rule. Per maintainer decision
("not production — clean breaks allowed, no migration scaffolding"), v0.3.0 may change derived keys,
ciphertext formats, and stored-record shapes **without** a migration path; affected accounts re-register.
There is no `kdfVersion`/WI-0 scaffold.

## 0. How to read this PRD

- This document covers **only what is NOT yet done.** Everything in §1 ("Already shipped") is closed and
  is listed for context, not action.
- Work items keep the prior numbering (WI-15, WI-16) so they trace back to the audit and the v0.2.2 PRD.
- Each WI gives a **proven problem** (with the current `tests/audit-*` proof and file:line), a **design**,
  the **files/deps** touched, and an **acceptance assertion** (the test to add or invert).
- §3 ("Developer-owned residuals") lists findings an SDK **structurally cannot fully fix**. They are not
  bugs to close in code; they are documented obligations on the integrator, with the optional affordances
  the SDK could add to help.

### WI-15 — Real server-side WebAuthn assertion (H7 / WEBAUTHN-1/2 / DESIGN-5, **High**)

**Problem (proven).** Biometric accounts do **not** prove a hardware WebAuthn assertion to the server.
Today the biometric flow (`registerWithBiometric` / `loginWithBiometric`, `src/client/authClient.ts`) is
**B1**: the client derives an ed25519 keypair from the PRF/gate secret (`src/client/authKey.ts`
`deriveAuthPublicKey` / `signAuthChallenge`) and the server verifies that *derived-key* signature
(`src/server/signature.ts:verifyAuthSignature`). The server never sees — and never verifies — a WebAuthn
assertion: no credential COSE public key is stored, and there is no check of `origin`, `rpIdHash`,
`signCount` (clone detection), or `userVerification`. Authentication therefore rests on a secret that
lives **client-side** (the PRF/gate secret), not on possession of the authenticator.

- *Current characterization:* `tests/audit-server.test.ts › WEBAUTHN-1` asserts only that login requires
  a *challenge signature* (not a bearer hash) — it does **not** assert a hardware assertion. That is the
  test to upgrade.
- *Severity:* High. The biometric factor is not actually bound to the device authenticator; anyone able
  to reproduce/exfiltrate the PRF or gate secret can mint a valid login signature off-device.

**Goal.** Make biometric authentication a **verified WebAuthn assertion**: registration verifies an
attestation and stores the credential's public key; login verifies an assertion signature against that
stored key, including `origin`, `rpIdHash`, `challenge`, `userVerification`, and an increasing
`signCount`. The PRF/gate secret becomes the **client-side encryption key only**, never an auth credential.

**Non-goals.**
- Email/passkey accounts (a *typed* passkey + PBKDF2 — **not** WebAuthn) keep ed25519 signature auth; WI-15
  does not touch them.
- Wallet (Solana) auth is unchanged.
- No backward migration of existing B1 biometric accounts — clean break, re-register (§ Migration).

**Design.**

1. **Dependencies.** Add `@simplewebauthn/server` (server-side verify) and `@simplewebauthn/browser`
   (client ceremony helpers), or implement equivalent verification. Server lib is a **dependency** of the
   `/server` entry; browser lib a **peer/optional** of `/client`. (Verify bundle-size + ESM compatibility
   with the existing tsup/Next setup.)

2. **Stored credential model.** Extend `UserData` (`src/core/types.ts`) with:
   ```ts
   webauthnCredentials?: Array<{
     credentialID: string;     // base64url
     publicKey: string;        // base64url COSE key
     signCount: number;        // monotonic; clone detection
     transports?: string[];
     createdAt: number;
   }>;
   ```
   `authPublicKey` (the B1 ed25519 key) is **removed for biometric accounts** (email accounts keep it).

3. **New endpoints / actions** (added to `src/server/routes.ts` + `src/next/routes.ts`):
   - `POST webauthn/register-options` → server issues `PublicKeyCredentialCreationOptions` (challenge,
     rp, user, pubKeyCredParams, `userVerification: "required"`, PRF extension) and stores the challenge.
   - `POST webauthn/register-verify` → `verifyRegistrationResponse`; on success store the credential
     (public key, `signCount`, transports) and issue a session.
   - `POST webauthn/login-options` → issues `PublicKeyCredentialRequestOptions` (challenge,
     `allowCredentials` from stored credentials, `userVerification: "required"`); stores the challenge.
   - `POST webauthn/login-verify` → `verifyAuthenticationResponse` against the stored credential public
     key; verify `origin`, `rpID`/`rpIdHash`, challenge match (single-use consume), and **`signCount`
     strictly increasing** (reject clones); update `signCount`; issue a session **only** on success.

4. **Client.** `registerWithBiometric` / `loginWithBiometric` (`src/client/authClient.ts`) drive the two
   ceremonies via `@simplewebauthn/browser` (`startRegistration` / `startAuthentication`). The PRF
   extension still yields the **encryption** secret used to derive/unwrap the app key client-side; the
   assertion is what authenticates. `src/client/authKey.ts` (`deriveAuthPublicKey` / `signAuthChallenge`)
   is **no longer used for biometric** (kept for email).

5. **Config.** `webauthn.rpId` becomes **required** for biometric in production (must match the
   registrable domain); document the `origin` allow-list. Surface a clear startup error if biometric is
   used without `rpId` configured.

**Files.** `src/core/types.ts`, `src/server/routes.ts`, `src/server/signature.ts` (or a new
`src/server/webauthn.ts`), `src/next/routes.ts`, `src/client/authClient.ts`, `src/client/webauthn.ts`,
`src/core/config.ts`, `package.json`.

**Migration.** Clean break. Existing B1 biometric accounts (which stored an `authPublicKey`) cannot
produce an assertion for a credential the server never registered → they **re-register** the biometric
factor. Email/wallet accounts are unaffected. Document in the demo migration notes.

**Acceptance / proof tests.**
- Upgrade `tests/audit-server.test.ts › WEBAUTHN-1`: a biometric login with only a derived-key signature
  (no assertion) is **rejected**; a verified assertion succeeds.
- New tests: a **replayed** assertion (reused challenge) is rejected; a **tampered** assertion (bad
  signature / wrong `origin` / wrong `rpIdHash`) is rejected; a **non-increasing `signCount`** is rejected
  (clone detection).
- `tsc` clean, full suite green.

**Risks / open questions.**
- `@simplewebauthn/*` adds dependency + bundle weight to a currently dep-light SDK — confirm acceptable.
- Test environment: WebAuthn ceremonies need mocking (no real authenticator in jest/node) — build a fake
  authenticator/assertion fixture, or unit-test the verify layer with `@simplewebauthn/server` test vectors.
- Decide: keep B1 as an **explicit opt-in** fallback for environments without a usable authenticator, or
  drop it entirely? (Recommendation: drop B1 for biometric; keep email/passkey for non-WebAuthn fallback.)

### WI-16 — HKDF enc/auth key separation + server-side passkey recovery (WEBAUTHN-3/5, Med)

**Problem (proven).** One secret is reused for two cryptographic purposes. The app key both **encrypts
wallets** and **seeds the auth keypair** (`src/client/authKey.ts:authSeed` = `SHA-256("ttc-auth-v1:" +
appKey)`). While the `"ttc-auth-v1:"` prefix is a form of domain separation, it is ad-hoc, and the same
high-entropy secret still does double duty. Separately, there is **no server-side recovery path** for a
WebAuthn credential: a cleared `localStorage` / new device leaves a biometric user stranded (the wrapped
unlock blob is per-device, `src/client/biometricUnlock.ts`).

**Goal.**
1. **Key separation:** derive purpose-specific subkeys via HKDF instead of reusing one secret —
   `encKey = HKDF(secret, info="ttc-enc-v1")`, `authValue = HKDF(secret, info="ttc-auth-v1")`.
2. **Recovery:** persist the data needed to re-establish access on a new device without weakening the
   model, and surface key-loss risk in the UI.

**Design.**
- **HKDF separation** (`src/core/crypto.ts`, `src/client/authKey.ts`): replace the bare
  `SHA-256("ttc-auth-v1:"+appKey)` seed and any direct double-use of the app key with
  `HKDF-SHA-256(secret, salt, info)` using distinct `info` labels for encryption vs. (where still
  applicable) authentication. For email accounts (which keep ed25519 signature auth) the auth seed becomes
  `HKDF(appKey, info="ttc-auth-v1")`; the encryption key is `HKDF(appKey, info="ttc-enc-v1")`. `@noble/hashes`
  already provides `hkdf` (`@noble/hashes/hkdf.js`) — no new dependency.
- **Recovery (depends on WI-15's credential model):**
  - Store **`credentialID` + the KDF `salt`** server-side (the salt is **not** secret and is useless
    without the authenticator). This lets a returning/new-device client fetch what it needs to re-derive
    and to populate `allowCredentials`.
  - Support **discoverable credentials** (resident keys, `residentKey: "preferred"`) so a user can
    authenticate on a fresh device/browser without prior local state.
  - **Surface key-loss risk in the UI** (and demo): a non-custodial wallet whose only factor is a single
    device authenticator is unrecoverable if that authenticator is lost — prompt users to register a
    second factor / backup.

**Files.** `src/core/crypto.ts`, `src/client/authKey.ts`, `src/client/biometricUnlock.ts`,
`src/server/routes.ts` + `src/server/session.ts` (persist salt/credentialID), `src/core/types.ts`, demo UI.

**Migration.** Changing the auth-seed derivation (bare SHA-256 → HKDF) changes the derived ed25519 key for
**email** accounts → clean break, re-register (consistent with §0). Bundle this with WI-15 so biometric +
email auth-model changes land in one breaking release.

**Acceptance / proof tests.**
- Unit: `encKey ≠ authValue` for the same input secret; both deterministic; HKDF output matches a known
  vector.
- Recovery: a client with **no local state** can complete a discoverable-credential login against a
  server that stored only `credentialID` + non-secret `salt` (mocked authenticator).
- `tsc` clean, full suite green.

### WI-20 — Per-user random PBKDF2 salt (grok; pre-leak precomputation resistance, Low–Med)

**Problem (proven).** The email/passkey app-key salt is **deterministic**:
`salt = SHA-256(appId : normalizedEmail)` (`src/core/crypto.ts:deriveAppKeyFromPasskey`). `appId` is a
per-deployment value that is typically public (often the domain), and the email is the target's own
address — so an attacker who knows both can **precompute** PBKDF2 candidates against a target *before* any
database leak. A per-user **random** salt removes that pre-leak window: precomputation can't begin until
the (post-leak) salt is known. Only the **email/passkey** path is affected — wallet keys are
`SHA-256(signature)` and biometric keys are PRF-derived, both high-entropy, so salt precomputation is moot
there.

- *Scope of benefit:* defense-in-depth for low-entropy user-chosen passkeys. With 600k iterations the
  per-candidate cost is already high; the random salt's marginal win is "no offline table can be built
  ahead of a leak, and no single table spans multiple deployments."

**Goal.** Replace the deterministic email salt with a **per-user random salt** (≥16 bytes), stored
server-side (not secret), without changing the deterministic-recovery guarantee.

**Design.**
- `deriveAppKeyFromPasskey(passkey, salt, iterations)` takes the salt explicitly (random hex), instead of
  computing it from the email.
- **Register:** client generates a random salt (`randomHex(16)`), derives the app key, and sends the salt
  alongside `authPublicKey` + `pbkdf2Iterations`. Server stores it on `UserData.pbkdf2Salt`.
- **Login:** the `POST /challenge` response already returns `pbkdf2Iterations`; it additionally returns
  `pbkdf2Salt`. The client re-derives the app key from `(passkey, pbkdf2Salt, pbkdf2Iterations)`. The salt
  is **not secret** — returning it is equivalent to returning the iteration count.
- **`appId` interaction:** a random per-registration salt *subsumes* `appId`'s role on the email path
  (two registrations — even same email+passkey on two apps — get different salts → different keys), so
  `appId` can be dropped from the email salt. **`appId` stays for the wallet path** (the appId-bound
  `walletAppKeyMessage`, WI-14), which has no random-salt equivalent.
- *Trade-off:* the app key is no longer derivable purely offline from `(passkey, email)` — it requires the
  server-stored salt. This is **not a new round-trip** (login already fetches a challenge), and the salt
  is as durable as the rest of the user record.

**Files.** `src/core/crypto.ts`, `src/client/authClient.ts`, `src/server/routes.ts`, `src/core/types.ts`
(`UserData.pbkdf2Salt`), tests.

**Migration.** Clean break (changes the email-account app key) → re-register. Bundle with WI-16 (which
already re-derives email keys via HKDF) so the email-account key change happens once.

**Acceptance / proof tests.**
- Register stores a random `pbkdf2Salt`; two accounts with the **same** email+passkey derive **different**
  app keys (random salts differ).
- `/challenge` returns `pbkdf2Salt`; a login that re-derives with the returned salt+iterations succeeds; a
  wrong/missing salt fails.
- `tsc` clean, suite green.

**Risk.** Modest benefit vs. added field + lost offline-derivation simplicity — confirm it's worth the
clean break (it pairs naturally with the WI-16 email-key change, so the breakage is shared).

### WI-21 — Auth-event hooks for observability (grok: audit logs, Low)

**Problem (proven).** The route handlers (`src/server/routes.ts`) emit **no observable events**. An
integrator cannot feed a SIEM, alert on brute-force, or build an auth audit trail without forking the
handlers. grok flagged "audit logs for auth events."

**Goal.** An optional, best-effort event callback on the route factory so integrators can log/alert/observe
without modifying the SDK — carrying **non-secret** identifiers only.

**Design.**
- Add `onAuthEvent?: (e: AuthEvent) => void | Promise<void>` to `AuthHandlerOptions`
  (`createAuthHandlers` / `createNextAuthRoutes`).
- `AuthEvent` (`src/core/types.ts`):
  ```ts
  type AuthEvent = {
    type: "register" | "login.success" | "login.failure" | "logout"
        | "challenge.issued" | "ratelimit.exceeded";
    method?: "email" | "wallet" | "biometric";
    publicKey?: string;     // public identifier only
    email?: string;         // integrator may choose to hash; the SDK passes it through
    ip?: string;            // as resolved by clientIp (only meaningful behind a trusted proxy)
    ts: number;
  };
  ```
- **Never** include secrets in events — no session tokens, no signatures, no passkeys, no app keys.
- **Best-effort, non-blocking, must not throw** — wrap in `try/catch` (mirror the existing
  `registerSessionClearHook` pattern in `src/client/session.ts`); a throwing or slow hook must not block or
  fail the request.
- Fire at: register, login success/failure, logout, challenge issuance, and rate-limit rejection.

**Files.** `src/server/routes.ts`, `src/core/types.ts`, `src/server/index.ts` (export the type),
`src/next/index.ts` (pass-through), tests.

**Migration.** Purely additive — no breaking change.

**Acceptance / proof tests.**
- `onAuthEvent` fires with the correct `type`/`method` on login success, login failure, and rate-limit
  rejection; the payload contains **no** token/signature/secret.
- A throwing `onAuthEvent` does **not** break the request (handler still returns its normal response).
- `tsc` clean, suite green.

**Risk.** PII in events (email/IP) — document that the integrator is responsible for redaction/retention;
the SDK passes identifiers through rather than hashing them (so the integrator controls the policy).

## 3. Developer-owned residuals (NOT SDK bugs — documented obligations)

These findings are **inherent to a client-side SDK** and cannot be fully closed in library code. They are
already documented in the demo migration PRD ([`tetrac-login-sdk-demo/audit/v0.3.md`](../../tetrac-login-sdk-demo/audit/v0.3.md)
§6/§8). Listed here so v0.3.0 sign-off is explicit about why they remain open.

| Finding | Why the SDK can't fully fix it | Integrator obligation | Optional SDK affordance |
|---|---|---|---|
| **SERVERSIDE-5** — email register has no proof of ownership | The SDK sends no email and can't verify control of an address | Verify email (magic link / OTP) **before** register; bot-gate (CAPTCHA/Turnstile) | A pre-register hook / `onBeforeRegister` callback the integrator can gate on |
| **CLIENTVAULT-2** — session token + email in `localStorage` (XSS-exfiltratable) | Bearer token must persist somewhere the page can read; email is the PBKDF2 salt | Strict CSP / Trusted Types / SRI; short `autoLockMs` | Option to keep the token in memory-only (forces re-login on reload) |
| **Per-target challenge DoS** | No reliable client IP without a trusted proxy; per-target limiting trades global lockout for targeted | `trustProxyHeaders` + edge per-IP rate limiting | (Shipped) `trustedProxyHops`; document required edge config |
| **Login work-amplification** | Verify-first (the AUTHSESSION-3 fix) means a flood does cheap work before the failure counter trips | Edge/volumetric rate limiting | None without re-introducing victim lockout |
| **Account enumeration via `/challenge`** | A real login needs a challenge for known accounts and can't issue one for unknown — structural | Email verification; edge limiting; optional decoy challenges | Optionally return a decoy challenge for unknown emails |

## 4. Release engineering (external — not code)

1. **`npm publish --provenance` from CI** — requires the CI workflow (shipped, `.github/workflows/ci.yml`)
   plus an npm automation token configured as a repo secret. Not doable in-repo.
2. **Demo dependency bump** — `tetrac-login-sdk-demo` pins `@tetrac/login-sdk@0.2.0` and its README says
   "≥ 0.2.0"; bump both to `^0.3.0` **after** `0.3.0` is published to npm (bumping earlier breaks the
   demo's install).
3. **`npm audit`** — 3 *moderate* advisories via `@solana/web3.js → jayson` (transitive peer, below the
   `--audit-level=high` gate). Track upstream Solana releases; not fixable in this repo.
4. **`CHANGELOG.md`** — author one for `0.3.0` enumerating the breaking defaults (memory-only vault,
   signature auth / no `passkeyHash`, AES-GCM ciphertext, `appId` required, 4h session TTL, `crypto-es`
   removed). The integrator-facing breaking-change list already exists in the demo's `audit/v0.3.md`.

## 5. Acceptance & rollout for v0.3.0

1. **WI-15 + WI-16 land together** (one breaking release) or WI-15 alone if WI-16 slips — both are clean
   breaks for biometric (and WI-16 for email) accounts; document re-registration.
2. **Proof tests inverted/added** per WI-15/WI-16; the existing 170-test baseline stays green; `tsc`

3. **Config/docs:** `webauthn.rpId` documented as required for biometric in production; key-loss/recovery
   guidance in the demo; `CHANGELOG.md` written.
4. **Publish:** `npm publish --provenance` from CI; then bump the demo.

### Suggested sequence
`WI-21 (event hooks — additive, ship anytime) → WI-15 (assertion + attestation) → WI-16 (HKDF
separation + recovery) + WI-20 (random salt) [bundle the email-key change once] → CHANGELOG → publish →
demo bump`.
WI-21 is non-breaking and can land independently at any point. WI-20 changes the email-account app key, so
bundle it with WI-16 (which already re-derives that key via HKDF) to take the clean break once. The
developer-owned residuals (§3) ship as documentation only; revisit if/when a trusted-proxy or
email-verification integration point is added.

## 6. Open questions

- **Drop B1 entirely, or keep as opt-in?** Recommendation: drop B1 for biometric (real assertion only);
  email/passkey remains the non-WebAuthn fallback.
- **`@simplewebauthn/*` as dependency vs. peer?** Server lib likely a dependency of `/server`; browser lib
  a peer of `/client`. Confirm bundle/ESM fit.
- **Is biometric a primary factor for v0.3.0 GA?** If yes, WI-15 is a release blocker (per the demo PRD
  note). If biometric stays "beta", v0.3.0 can ship with B1 clearly labelled and WI-15 tracked for 0.3.1.
