# v0.2.1 — Security Hardening PRD

**Based on:** DeepSeek V4 audit + empirical test results (133 tests, 9 new hardening suites)  
**Scope:** Core cryptography, storage-layer integrity, input validation, safe defaults  
**Non-goal:** No API/interface breaking changes. All migrations are additive with backward compat.

---

## Terminology Context

Before listing changes, a clarification that affects prioritization:

- **EVM wallets** are generated client-side, AES-encrypted under the app key, and stored as ciphertext alongside Solana wallets. They are used internally via `useEvmSigner` (viem `LocalAccount`). They are **never used for Web3 login/registration** — no `verifyEvmSignature` is needed because users do not authenticate with EVM addresses.
- **Solana wallets** are the sole Web3 auth mechanism. The server correctly verifies Solana signatures (`verifySolanaSignature` using `tweetnacl`).
- The `publicKey` field in `UserData` is the Solana public key (or the email/biometric identity key), not an EVM address.

F7 from the audit (EVM signature verification) is therefore **not a vulnerability** — removing it from the hardening roadmap.

## Change 1: AES-GCM Authenticated Encryption (Critical)

### Problem
`encryptSecret` / `decryptSecret` use `CryptoES.AES.encrypt` which is OpenSSL-EVP-KDF + AES-CBC with **no authentication tag**. Attackers with KV store write access can silently corrupt wallet secrets or trigger "wrong key" errors indistinguishable from real auth failures.

**Test evidence** (`tests/ciphertext-tampering.test.ts`):
- 2/128 single-bitflip positions → silent garbage returned (no error)
- 126/128 bitflips → `"Decryption failed: wrong key or corrupted ciphertext"` — same message as entering wrong passkey
- `CryptoES.AES.decrypt` itself never throws; only the UTF-8 encoding conversion may fail

### Specification

**New functions in `src/core/crypto.ts`:**

```typescript
// AES-256-GCM encrypt. Returns "v2:${b64url(iv)}:${b64url(ciphertext + auth_tag)}"
async function encryptSecretV2(plaintext: string, appKey: string): Promise<string>

// AES-256-GCM decrypt. Throws on any tampering (GCM auth tag mismatch).
async function decryptSecretV2(ciphertext: string, appKey: string): Promise<string>
```

**Format:**
- Prefix: `v2:` (distinct from v1 which has no prefix)
- IV: 96-bit random, base64url-encoded
- Ciphertext + 128-bit GCM auth tag: base64url-encoded
- Full string: `v2:${iv_b64url}:${ct_b64url}`

**Implementation constraints:**
- Must use `crypto.subtle` (Web Crypto API), not `crypto-es`
- IV must be generated via `crypto.getRandomValues` (CSPRNG)
- Key is imported as `raw` from the existing hex app key (no change to key derivation)
- `extractable: false` is NOT needed here (the app key is already in JS memory)
- Must work in both browser and Node 18+ (Edge Runtime compatible)

**Migration path:**
- `encryptSecret` (v1) kept for decrypting existing ciphertext
- `decryptSecret` (v1) kept; `decryptSecretV2` added alongside
- All NEW wallet generation (`generateWalletBundle`, `registerWithEmail`, `registerWithBiometric`, `connectWallet`) uses v2
- During transition: `decryptSecret` wrapper tries v2 first, falls back to v1
- After v1.0: remove v1 functions, document migration

**Wallet layer changes (`src/client/wallet.ts`):**
- `generateSolanaWallet` / `generateEvmWallet`: call `encryptSecretV2` instead of `encryptSecret`
- `decryptWalletSecret`: call `decryptSecretV2` with v1 fallback
- `toSolanaKeypair` / `withDecryptedKey`: updated automatically via `decryptWalletSecret`

**Config additions (`src/core/config.ts`):**
```typescript
export interface AuthConfig {
  // ...
  /** Cipher version for new encryption: "v2" (AES-GCM) or "v1" (AES-CBC, legacy) */
  cipherVersion: "v1" | "v2";
}
```
Default: `"v2"`. Deployers can set to `"v1"` during migration if they must maintain byte-exact compat with very old clients.

### Backward Compatibility
- V1 ciphertext can always be decrypted (no format change to stored data)
- V2 ciphertext is longer (~20% overhead) but fits within the existing `encryptedSecret` field limit (8192 chars)
- `flattenBundle` / `EncryptedWallet` type unchanged

### Test Verification
- `encryptSecretV2` + `decryptSecretV2` round-trip: 100% pass
- Tampered ciphertext (bitflip at every byte position): 100% throw `OperationError`
- Tampered IV: throw
- Truncated ciphertext: throw
- Wrong key: throw
- V1 ciphertext still decrypts via fallback path

## Change 2: PBKDF2 Iteration Default Bump (High)

### Problem
Default is **100,000** iterations. OWASP 2023 guidance: **600,000** for PBKDF2-HMAC-SHA256. Offline attackers brute-force passkeys ~6x faster than current best practices.

**Test evidence** (`tests/pbkdf2-iteration-cost.test.ts`):
- 100k: ~1,200ms (Apple M-series)
- 600k: ~7,200ms
- 1M: ~12,000ms
- Different iteration counts produce different keys (safe for migration)
- Cross-iteration decrypt fails — proving existing users are locked to their creation-time count

### Specification

**Single change in `src/core/config.ts`:**

```typescript
export const DEFAULT_CONFIG: AuthConfig = {
  // ...
  pbkdf2Iterations: 600_000,  // was 100_000
  // ...
};
```

**Why this is safe:**
- `deriveAppKeyFromPasskey(passkey, email, iterations)` is a pure function
- Existing users' wallets were encrypted at their creation-time iteration count
- On login, the client re-derives the app key using the iteration count from the *current* config
- If an existing user's `encryptedSecret` was created at 100k, the client at 600k derives a **different key** and `decryptSecret` fails
- **Fix:** The iteration count must be stored per-user on the server, OR the client must know which iteration count to use

**Alternative approaches (choose one):**

**Option A (recommended):** Store iteration count in `UserData` on the server.
```typescript
export interface UserData {
  // ...
  pbkdf2Iterations?: number;  // iterations used at account creation
}
```
- On `register`: store `config.pbkdf2Iterations`
- On `login`: server returns `pbkdf2Iterations` in the `UserData` blob
- Client uses the stored count, not config default
- Future bumps only affect NEW registrations

**Option B (simpler):** Write a deterministic version tag into the app key derivation.
- Actually more complex. Don't do this.

**Option C (simplest, but wastes cycles):** Always try default iterations first, fall back to common older values.
- Not recommended — 7s delay per fallback attempt is bad UX.

**Recommendation: Option A.** Add `pbkdf2Iterations` to `UserData` on registration, return it on login, pass it through to `deriveAppKeyFromPasskey`. Config default is now 600,000.

### Config
- `pbkdf2Iterations`: 600,000 (default), range [100k–2M]
- Deployers with existing users on 100k must set back to 100k until all users have re-registered at the new count, OR implement Option A

## Change 3: Salted Passkey Verification Hash (High)

### Problem
`hashPasskey(passkey)` = `SHA-256(passkey)` — no salt, no work factor.
- Same passkey → identical hash across all users (rainbow table vulnerability)
- ~100K hashes/second in Node.js — 8.6B passwords/day per core
- No algorithmic agility for future migration

**Test evidence** (`tests/salted-hash.test.ts`):
- Two different users with `passkey="snowflake2024"` → identical stored `passkeyHash`
- SHA-256 throughput benchmark: ~100K hashes/sec
- 128-bit salt space: 2^128 ≈ 10^39 values (precomputation infeasible)

### Specification

**New hash format (server-side, stored in `UserData.passkeyHash`):**

```typescript
export interface PasskeyHashV2 {
  algorithm: "pbkdf2-sha256-v2";
  /** 128-bit random salt, hex-encoded (32 hex chars) */
  salt: string;
  /** PBKDF2 output, hex-encoded (64 hex chars) */
  hash: string;
  /** Iterations used (default 600_000) */
  iterations: number;
}
```

Serialized as JSON string: `'{"algorithm":"pbkdf2-sha256-v2","salt":"<32hex>","hash":"<64hex>","iterations":600000}'`

**New functions in `src/core/crypto.ts`:**

```typescript
async function hashPasskeyV2(passkey: string): Promise<string>  // returns JSON string
async function verifyPasskeyV2(passkey: string, stored: string): Promise<boolean>  // stored = JSON string
```

**Implementation:**
- `hashPasskeyV2`: generate 16 random bytes via `crypto.getRandomValues`, PBKDF2(passkey, salt, 600k, SHA-256), return JSON
- `verifyPasskeyV2`: parse stored JSON, extract salt+iterations, recompute PBKDF2, compare via `timingSafeEqual`
- Both use `crypto.subtle.importKey("raw", ...)" + "deriveBits"` (Web Crypto API)

**Server changes (`src/server/routes.ts`):**
- On `register`: accept either v1 (64-char hex) or v2 (JSON string) in `passkeyHash`
- On `login`: detect format — if v2, call `verifyPasskeyV2`; if v1, use existing `timingSafeEqual` compare
- Store format in `UserData` as-is (string field, no type change)

**Migration:**
- Existing users with v1 hashes continue to work (no re-hash needed)
- New registrations use v2
- After v1.0: deprecate v1 acceptance, require v2 for new accounts
- The `trustProxyHeaders`-type config: `passkeyHashAlgorithm: "v2" | "v1"` (default `"v2"`)

### Backward Compatibility
- V1 `passkeyHash` (64-char hex) still accepted on login
- V2 hash is a JSON string — server detects format by checking `startsWith("{")` vs `/[0-9a-f]{64}/`
- Old clients sending v1 hashes for new registrations are rejected if server is in v2 mode
- New clients sending v2 hashes are accepted by both old and new servers (old server treats it as opaque string and compares via `timingSafeEqual` — won't match a v1 hash, so login will fail, but registration stores it correctly)

## Change 4: Input Validation Hardening (Medium)

### Problem
Several string fields accept arbitrary values with no format or length validation.

**Test evidence** (`tests/input-validation.test.ts`):
- Empty email `""` → accepted (201)
- XSS payload `"javascript:alert(1)@foo.com"` → accepted (201)
- 10,000-char email → accepted (201)
- Non-hex `passkeyHash` `"this is NOT hex!!!"` → accepted (201)
- 6-char `passkeyHash` (not 64-char) → accepted (201)
- Whitespace-only `publicKey` `"   \n\t  "` → accepted (201)
- 100,000-char `publicKey` → accepted (201)

### Specification

**Validators in `src/server/routes.ts`:**

```typescript
const EMAIL_RE = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
const HEX64_RE = /^[0-9a-f]{64}$/i;
const SOLANA_PK_RE = /^[1-9A-HJ-NP-Za-km-z]{32,44}$/;  // base58 approx

function validateEmail(email: string): string | null {
  if (!EMAIL_RE.test(email)) return "Invalid email format";
  if (email.length > 320) return "Email too long";
  return null;
}

function validatePasskeyHash(hash: string): string | null {
  // v1: 64-char hex
  if (HEX64_RE.test(hash)) return null;
  // v2: JSON object
  try {
    const parsed = JSON.parse(hash);
    if (parsed?.algorithm === "pbkdf2-sha256-v2" && typeof parsed.salt === "string" && typeof parsed.hash === "string") {
      return null;
    }
  } catch { /* not JSON */ }
  return "Invalid passkeyHash format";
}

function validatePublicKey(key: string): string | null {
  if (!key || key.trim() !== key) return "Invalid publicKey format";
  if (key.length > 128) return "publicKey too long";
  return null;
}
```

**Integration points:**
- `register`: validate `email`, `publicKey`, `passkeyHash` before any DB operations
- `login`: validate `email`, `passkeyHash`
- `challenge`: validate `publicKey`
- `connect-wallet` / `login-wallet`: validate `publicKey`
- `validateWallets`: add `publicKey` length check per entry (currently only checks existence)

**Rejection behavior:**
- Return 400 with `{ error: "Invalid email format" }` — field-specific errors are safe because these are request validation, not credential checks
- Do NOT distinguish "email format invalid" from "email not found" on login (already handled: login returns generic "Invalid credentials")

## Change 5: Safer Default for App Key Storage (Medium)

### Problem
Default `appKeyStorage: "session"` persists the encryption key in `sessionStorage`. Any XSS on the origin can read `sessionStorage.getItem("ttc_ek")` and decrypt all wallet secrets offline.

**No new test added** — existing `tests/vault-signer.test.ts` confirms the in-memory vault works correctly.

### Specification

**Single change in `src/core/config.ts`:**

```typescript
export const DEFAULT_CONFIG: AuthConfig = {
  // ...
  appKeyStorage: "memory",  // was "session"
  // ...
};
```

**Behavior change:**
- Default: key lives only in memory → tab reload forces re-auth
- Users who want session-persistence across reloads opt-in: `appKeyStorage: "session"`
- The `"memory"` mode is already implemented and tested
- `lockOnHide: true` remains default (lock on tab hide regardless of storage mode)

**Documentation note for README:**
> The app (encryption) key defaults to in-memory only. On tab reload, the vault locks and the user must re-authenticate to decrypt wallets. Set `appKeyStorage: "session"` in `AuthConfig` to persist across reloads within the same origin.

## Change 6: Remove Invalid EVM-Finding From Codebase (Informational)

No code change needed. The EVM wallets are internal signing wallets generated client-side. The `verifySolanaSignature` check in the server routes is correct — only Solana wallets are used for Web3 authentication. The EVM test suite (`tests/evm-verification.test.ts`) should be **updated** to reflect this, not removed — it serves as documentation that the behavior is intentional.

**Update `tests/evm-verification.test.ts`:** Change the "vulnerability" framing to "design intent documentation". Replace the "exploit" test with a test proving that EVM wallets are correctly generated and that the auth path correctly ignores them (because they are not used for authentication).

## Migration Guide for Deployers

### Upgrading from v0.1.x / v0.2.0

All changes are backward-compatible. Deployers can upgrade server and client independently.

| Change | Old client + new server | New client + old server |
|---|---|---|
| AES-GCM (v2 cipher) | Receives v1 ciphertext → uses v1 decrypt = OK | Sends v2 ciphertext → old server stores it → new client decrypts = OK |
| PBKDF2 600k default | New registrations use 600k. Old registrations unchanged. | Same behavior |
| Salted passkey hash | Old v1 hash accepted on login. | New v2 hash stored; old server treats as opaque string = OK |
| Input validation | Some previously-accepted inputs now get 400. | Already sending valid data = no change |
| `appKeyStorage: "memory"` | Tab reload no longer restores session. | Opt-in to session explicitly if needed |

### Rollout Order

1. **Server first:** Deploy new `routes.ts` with both v1 and v2 acceptance for `passkeyHash`, input validators
2. **Client update:** Deploy new SDK with AES-GCM v2 writers, salted hash v2 writers, memory-mode default
3. **Migration period:** Monitor for clients still sending v1 data; both formats accepted
4. **v1.0 cleanup:** Remove v1 CryptoES encrypt; require v2 hash format for new accounts
