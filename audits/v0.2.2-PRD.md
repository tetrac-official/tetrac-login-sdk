# PRD ‚Äî `@tetrac/login-sdk` v0.2.2 Security Hardening

**Target release:** `v0.2.2` (current `package.json` is `0.2.0`)
**Basis:** [`audits/opus-4.8-comprehensive-audit.md`](./opus-4.8-comprehensive-audit.md), empirically confirmed by the characterization suites `tests/audit-crypto.test.ts`, `tests/audit-server.test.ts`, `tests/audit-client-vault.test.ts` (29 tests) + deepseek's suites (`tests/ciphertext-tampering.test.ts`, etc.). Full suite green at authoring time: **162/162**.
**Supersedes:** the pre-existing deepseek audit/PRD (created before the opus-4.8 audit). This PRD bumps the agreed fixes into a single `v0.2.2`.

---

## 0. How to read this PRD

- Work items (WI) are **ordered**; later items may depend on earlier ones (notably **WI-0 versioning** unblocks several crypto changes).
- Every WI lists a **proof test** (the `audit-*` test that currently passes against the *vulnerable* code) and the **acceptance assertion** it must be **inverted** to after the fix. "Inverting" the green baseline is the definition of done.
- **Backward-compat rule:** no change may make existing stored wallets undecryptable or existing email/biometric/Solana logins fail. Anything that would change a derived key, a hash, or a ciphertext byte format MUST be **versioned** (WI-0) and migrated lazily (decrypt/verify under the stored version ‚Üí re-write under the current version on next unlock/login).
- **Out of scope (withdrawn, by design):** "EVM external wallet login." EVM keypairs are *internal client-generated signing wallets* (`useEvmSigner` ‚Üí `viem` `LocalAccount`); external wallet **login is Solana-only by design**, so `verifySolanaSignature` being ed25519-only is correct. The only EVM-related action is a **doc-clarity** note (WI-13). See the EVM scope correction in the audit.

## Phase 0 ‚Äî Versioning scaffold (do first)

### WI-0 ‚Äî Crypto-version metadata on stored records (DESIGN-9)
**Why first:** WI-1, WI-2, WI-12, WI-14, WI-17 all change a hash/key/ciphertext format and therefore need a per-record version to migrate without a flag-day.

**Change**
- Add optional version fields (all default to "v1/legacy" semantics when absent):
  - `EncryptedWallet`: `v?: number` (cipher scheme), `mac?: string` (see WI-2).
  - `UserData`: `kdfVersion?: number`, `pbkdf2Iterations?: number`, `hashVersion?: number`, `passkeySalt?: string` (see WI-1/WI-12).
- `decryptSecret`/`hashPasskey`-verify/`deriveAppKey` dispatch on the stored version; absence ‚áí legacy path.
- Add a single **`migrateOnUnlock`** helper (client) and **`migrateOnLogin`** (server) that, after a successful legacy decrypt/verify, re-writes the record under the current version.

**Files:** `src/core/types.ts`, `src/core/crypto.ts`, `src/server/session.ts`, `src/server/routes.ts`, `src/client/authClient.ts`, `src/client/wallet.ts`.

**Backward-compat:** purely additive; legacy records (no version fields) behave exactly as today.

**Acceptance:** new unit tests ‚Äî a record with no version fields decrypts/verifies via the legacy path; a record with `v:2`/`hashVersion:2` dispatches to the new path; round-trip both. No existing test changes.

## Phase 1 ‚Äî Backward-compatible hardening (ship in v0.2.2)

### WI-1 ‚Äî Server-side salted + peppered slow passkey verifier (C1, **Critical**)
**Problem (proven):** `tests/audit-crypto.test.ts` cracks a leaked `passkeyHash` from a 4-word dictionary and decrypts the wallet ‚Äî the stored verifier is unsalted single SHA-256.

**Change (no client/protocol change ‚Äî the client still sends `SHA-256(passkey)`):**
- Server stores `{ hashVersion:2, passkeySalt, pbkdf2Iterations, passkeyHash }` where
  `passkeyHash = PBKDF2-HMAC-SHA256(clientPasskeyHash, passkeySalt ‚à• SERVER_PASSKEY_PEPPER, ‚â•600k)`.
- New env: **`SERVER_PASSKEY_PEPPER`** (required in production; a DB-only leak without it cannot be cracked).
- Login: if `hashVersion` absent ‚Üí compare legacy bare SHA-256 (`timingSafeEqual`), and on success **upgrade-on-login** to v2. If `hashVersion:2` ‚Üí recompute & compare.
- Gate to `authMethod==='email'` and `'biometric'` (biometric `appKey` is already 256-bit, but salting is free and uniform). Reference patch: audit ¬ß"Patched code" `deriveStoredPasskeyHash`/`verifyStoredPasskeyHash`.

**Files:** `src/core/crypto.ts`, `src/server/routes.ts` (register/login), `src/core/config.ts` (read pepper), `src/core/types.ts`.

**Proof test ‚Üí invert:** `audit-crypto.test.ts ‚Ä∫ "END-TO-END ‚Ä¶ crackable offline"` ‚Äî after the fix, add `tests/audit-crypto.test.ts` (or a new `salted-verifier.test.ts`) asserting: (a) two users with the same passkey get **different** stored verifiers (random salt); (b) a dictionary attack on the *stored* value fails without the pepper; (c) a legacy bare-SHA-256 record still logs in and is upgraded.

### WI-2 ‚Äî Wallet-ciphertext integrity (H1, **High**) ‚Äî interim Encrypt-then-MAC
**Problem (proven):** `tests/ciphertext-tampering.test.ts` ‚Äî `CryptoES.AES.decrypt` of a tampered blob returns a malformed WordArray (negative `sigBytes`) with no error; `audit-crypto.test.ts` shows tampering is indistinguishable from a wrong key.

**Change (compat-preserving ‚Äî keeps `encryptedSecret` bytes intact for next-ttc reads):**
- On encrypt (client), also compute `mac = HMAC-SHA256(macSubkey(appKey), encryptedSecret)` and store it in the new `EncryptedWallet.mac` field (WI-0). `macSubkey = HMAC-SHA256("ttc-mac-v1", appKey)`.
- On decrypt (client), if `mac` present ‚Üí verify (`timingSafeEqual`) **before** AES-decrypt; on mismatch throw a **distinct** `CiphertextIntegrityError`. If `mac` absent (legacy) ‚Üí decrypt as today and re-write with a `mac` on next unlock.
- `validateWallets` must accept the optional `mac` string (bounded length).

**Files:** `src/core/crypto.ts`, `src/core/types.ts`, `src/client/wallet.ts`, `src/server/routes.ts` (`validateWallets`).

**Proof test ‚Üí invert:** `ciphertext-tampering.test.ts` / `audit-crypto.test.ts ‚Ä∫ "tampering is INDISTINGUISHABLE"` ‚Äî after the fix, a tampered blob **with a mac** must throw `CiphertextIntegrityError` (distinct from wrong-key). Keep a legacy-no-mac case asserting it still decrypts.

> Long-term replacement = WI-17 (AES-GCM). WI-2 buys integrity now without breaking next-ttc byte-compat.

### WI-3 ‚Äî Default `appKeyStorage:"memory"` (H3, **High**)
**Problem (proven):** `audit-client-vault.test.ts` reads the raw master `appKey` from `sessionStorage["ttc_ek"]`; default is `"session"`.

**Change:** `DEFAULT_CONFIG.appKeyStorage = "memory"`. Document the reload‚Üíre-auth trade-off; keep `"session"` available as an explicit, documented reduced-security opt-in (and, optionally, wrap the at-rest copy under a non-extractable Web Crypto key in IndexedDB like gate-mode).

**Files:** `src/core/config.ts`, README.

**Proof test ‚Üí invert:** `audit-client-vault.test.ts ‚Ä∫ H3` ‚Äî after the fix, the **default** config must NOT write `ttc_ek` to `sessionStorage`; only explicit `"session"` mode does. (UX note: surfaces a re-auth on reload ‚Äî covered by docs.)

### WI-4 ‚Äî Rate limiting: per-target buckets + real IP fallback (H5, **High**)
**Problem (proven):** `audit-server.test.ts ‚Ä∫ H5` ‚Äî three distinct clients/targets share one `ratelimit:unknown` bucket; `/challenge` & `/search-wallet` have no per-identity bucket.

**Change:**
- `/challenge` and `/search-wallet`: add a per-`publicKey` identifier so abuse is scoped per target.
- When `trustProxyHeaders` is false, derive a best-effort per-connection IP from the runtime (e.g. Next.js `request.ip`) instead of the constant `"unknown"`; only fall back to a global bucket with a **much higher** global ceiling distinct from the per-client limit.
- Keep `trustProxyHeaders` default false; document that production behind a trusted proxy should set it true (and see WI-4b).

**Files:** `src/server/http.ts` (`clientIp`), `src/server/routes.ts` (`rateLimited`, challenge/searchWallet), `src/core/config.ts` (global-ceiling option).

**Proof test ‚Üí invert:** `audit-server.test.ts ‚Ä∫ H5` ‚Äî after the fix, exhausting `/challenge` for `publicKey:"AAA"` must NOT 429 `/challenge` for `publicKey:"BBB"`.

**WI-4b (AUTHSESSION-4, Med):** when `trustProxyHeaders` is true, take the rightmost hop after a configurable trusted-hop count (`trustedProxyHops`) instead of the leftmost (spoofable) XFF value.

### WI-5 ‚Äî Penalty-only rate counter (AUTHSESSION-3, Med)
**Problem (proven):** `audit-server.test.ts ‚Ä∫ AUTHSESSION-3` ‚Äî the per-email counter increments **before** the credential check, so an attacker who knows the victim's email locks out the victim's own correct login.

**Change:** increment the per-identifier counter only on **failed** auth (or use a separate, higher success ceiling). Prefer tying lockout to source IP+identifier so victim lockout needs a botnet, not one host.

**Files:** `src/server/routes.ts` (login/loginWallet ordering), `src/server/rateLimit.ts` (optional `onFailureOnly` mode).

**Proof test ‚Üí invert:** `audit-server.test.ts ‚Ä∫ AUTHSESSION-3` ‚Äî after the fix, N failed logins for an email must NOT block that account's subsequent **correct** login.

---

### WI-6 ‚Äî Input validation + disjoint key namespaces + guarded parse (SERVERSIDE-1/8/11, AUTHSESSION-9, Med)
**Problem (proven):** `audit-server.test.ts ‚Ä∫ SERVERSIDE-1/8` (unauthenticated write into `pubKey:session:*`; `JSON.parse` crash) and `‚Ä∫ SERVERSIDE-11` (empty email, non-hex hash, whitespace/100k-char publicKey accepted).

**Change:**
- Validate `publicKey` (Solana base58 32-byte, or `0x`+40-hex for an internal EVM identity where applicable), `email` format, and `passkeyHash` (hex, length) before use. Reject `:`/control chars in keys. (This also cleanly rejects a stray `0x` on the Solana wallet-auth path with a clear error.)
- Move sessions to a **disjoint** namespace: `session:{token}` (not `pubKey:session:{token}`); user records `user:{publicKey}` (or keep `pubKey:` but ensure validated keys can never contain `session:`). Note: changing the session key format **invalidates existing sessions on deploy** ‚Äî users re-login (acceptable; document).
- Guard `getUserByPublicKey`'s `JSON.parse` (return `null` on parse error instead of throwing).
- Bound string-field lengths (publicKey, email, passkeyHash).

**Files:** `src/server/routes.ts`, `src/server/session.ts`, `src/core/config.ts` (prefixes), new `src/server/validate.ts`.

**Proof test ‚Üí invert:** `audit-server.test.ts ‚Ä∫ SERVERSIDE-1/8` and `‚Ä∫ SERVERSIDE-11` ‚Äî after the fix, registering `publicKey:"session:‚Ä¶"`/whitespace/100k-char/empty-email must return **400**, and no path may throw an unhandled error.

### WI-7 ‚Äî `import-wallet` total cap (SERVERSIDE-4, Med)
**Problem (proven):** `audit-server.test.ts ‚Ä∫ SERVERSIDE-4` grows a record from 1‚Üí81 wallets unbounded.

**Change:** add `maxWalletsPerUser` (default 32‚Äì64) enforced on the *resulting* total in `importWallet` and `connectWallet`; also bound the serialized record size before `persistUser`.

**Files:** `src/server/routes.ts`, `src/core/config.ts`.

**Proof test ‚Üí invert:** `audit-server.test.ts ‚Ä∫ SERVERSIDE-4` ‚Äî after the fix, exceeding the cap returns **400** and the total stops growing.

### WI-8 ‚Äî Persisted absolute lock deadline + lifecycle handlers (CLIENTVAULT-3/4, Med)
**Problem (proven):** `audit-client-vault.test.ts ‚Ä∫ CLIENTVAULT-3` ‚Äî only the key (no expiry) is persisted, so a reload re-hydrates **unlocked** with a fresh window regardless of elapsed idle time.

**Change:** persist an **absolute** deadline (e.g. `ttc_ek_exp`) alongside the key; on re-hydrate, if `now > storedDeadline` call `clearKeyState()` instead of arming a fresh window. Add `pagehide` + Page-Lifecycle `freeze` handlers that lock, and treat `pageshow` with `event.persisted` as a forced re-lock. (Mostly moot once WI-3 defaults to `"memory"`, but correct for explicit `"session"` users.)

**Files:** `src/client/session.ts`.

**Proof test ‚Üí invert:** `audit-client-vault.test.ts ‚Ä∫ CLIENTVAULT-3 "reloaded tab re-hydrates UNLOCKED"` ‚Äî after the fix, a reload past the stored deadline must report **locked** (`isLocked()===true`, `getAppKey()===null`).

### WI-12 ‚Äî 600k PBKDF2 + domain-separated salt for NEW accounts (H2, **High**, versioned)
**Problem (proven):** `audit-crypto.test.ts ‚Ä∫ H2` ‚Äî default 100k, salt is the bare email, no domain separation.

**Change (requires WI-0 to persist params per user):**
- New accounts: `kdfVersion:2`, `pbkdf2Iterations:600_000`, salt = `SHA-256(appId ‚à• ':' ‚à• normalizedEmail)` (deterministic ‚áí recovery preserved; domain-separated ‚áí not the bare email, unique per deployment). New `appId` config (required).
- Persist `kdfVersion` + `pbkdf2Iterations` in `UserData`; existing 100k/email-salt accounts keep deriving correctly. Migrate lazily on next login (re-derive under v2 + re-encrypt wallets).
- Reference patch: audit ¬ß"Patched code" `deriveAppKeyFromPasskey(..., kdfVersion)` + `deriveKdfSalt`.

**Files:** `src/core/crypto.ts`, `src/core/config.ts`, `src/client/authClient.ts`, `src/server/session.ts`/`routes.ts` (persist params), `src/core/types.ts`.

**Proof test ‚Üí invert:** `audit-crypto.test.ts ‚Ä∫ H2` ‚Äî after the fix, `DEFAULT_CONFIG.pbkdf2Iterations === 600_000`; a v2 account's salt is NOT the bare email; a legacy v1 account still derives the same key it did at 100k.

## Phase 2 ‚Äî Versioned / auth-model changes (v0.2.2 if time allows, else v0.3.0)

### WI-14 ‚Äî Domain-bound Solana key-derivation message (H4 / DESIGN-1, **High**, versioned)
**Problem (proven):** `audit-crypto.test.ts ‚Ä∫ H4` ‚Äî `WALLET_APP_KEY_MESSAGE` is a fixed constant with no origin; any site that gets the Solana wallet to sign it derives the same `appKey`.

**Change:** bind the message to the origin/`appId` (structured, wallet-rendered). Gate behind `kdfVersion` (WI-0): new accounts use the bound message; legacy accounts keep the constant for decrypt and migrate on unlock. Use `HKDF(sig, info)` rather than bare `SHA-256` to allow future per-origin subkeys.

**Proof test ‚Üí invert:** `audit-crypto.test.ts ‚Ä∫ H4` ‚Äî after the fix, the message includes a domain/appId and two different origins derive **different** keys.

### WI-15 ‚Äî Real server-side WebAuthn (H7 / WEBAUTHN-1/2 / DESIGN-5, **High**)
**Problem (proven):** `audit-server.test.ts ‚Ä∫ WEBAUTHN-1` ‚Äî biometric "login" is a hash compare with no assertion.

**Change:** add a server challenge endpoint; verify attestation at registration and **store the credential COSE public key**; verify the assertion signature + origin + rpIdHash + challenge + counter on login (e.g. `@simplewebauthn/server`). Gate session issuance on the verified assertion, not on `SHA-256(appKey)`. Treat the PRF/gate secret as the client-side *encryption* key only.

**Proof test ‚Üí invert:** `audit-server.test.ts ‚Ä∫ WEBAUTHN-1` ‚Äî after the fix, a login with only `{email, passkeyHash}` and no assertion must be **rejected**; a tampered/replayed assertion must be rejected.

### WI-16 ‚Äî HKDF enc/auth separation + server-side passkey recovery (WEBAUTHN-3/5, Med)
- `encKey = HKDF(prf, "ttc-enc-v1")`, `authValue = HKDF(prf, "ttc-auth-v1")` ‚Äî never reuse one secret for both.
- Store `salt`+`credentialId` server-side (salt is not secret, useless without the authenticator); support discoverable credentials so a cleared `localStorage` / new device is recoverable. Surface key-loss risk in UI.

## Withdrawn / out of scope (do **not** implement)
- **EVM external wallet login / EVM signature verification** ‚Äî by design, external auth is Solana-only; EVM is internal-signing-only. Only the WI-13 doc note applies.
- **CRYPTO-5 (timingSafeEqual)** ‚Äî disproven; the implementation is correct for fixed-length hex. Optional server-side `crypto.timingSafeEqual` is a nicety, not required.
- **SUPPLYCHAIN-3 (uuid CVE)** ‚Äî false positive; `npm audit` is clean.
